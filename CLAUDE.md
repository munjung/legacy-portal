# CLAUDE.md - legacy-portal

KT DS 「클린코드 & 리팩토링 실전」 과정의 관통 실습 프로젝트 (사내 업무 포털 - 결재/공지/일정).
이 파일은 **프로젝트의 사실(Facts)과 항상 지키는 코딩 규칙**만 담는다.
리팩토링 절차와 작업 방법은 `.claude/skills/refactor` Skill에서 관리한다.

## 빌드 / 실행 / 테스트
- 실행: `.\mvnw.cmd spring-boot:run`  (포트 8080)
- 테스트: `.\mvnw.cmd test`
- DB: MariaDB 있으면 사용, 없으면 자동 H2 인메모리
- Java 21 · Spring Boot 3.2 · JPA · JUnit 5

## 패키지 구조
- `approval`(결재) · `notice`(공지) · `schedule`(일정) · `user`(사용자) = 도메인(업무 폴더)
- 각 도메인 폴더 안 = Entity + Repository + Service (+ Controller)
- `common` = 공통 (메일·로그)

## 네이밍 컨벤션 (항상 지킴)
- 이름 = 약어·한 글자 금지, 의미 있는 완전한 이름
- boolean = `is` / `has` 접두

## 리팩토링 - 개선 방향 (내부 구조를 이렇게 바꾼다)
- 매직넘버 → 의미 있는 enum (단 DB 저장값은 정수 그대로 - 불변 규칙 참조)
- 직접 `new` → 생성자 주입 (DIP)
- Entity를 API에 직접 노출 → 요청/응답을 DTO(record)로 분리 (JSON 구조·필드는 레거시와 동일)
- 긴 메서드 분리 · 중복 제거 · 로직을 도메인으로 이동(Rich Domain)

## 리팩토링 - 불변 규칙 (외부에서 관찰 가능한 것은 절대 안 바꿈)
리팩토링 = **내부 구조만 개선, 겉보기 동작·계약은 100% 보존**. 아래는 절대 바꾸지 않는다.
- **API 엔드포인트 변경 금지** - 기존 URL·HTTP 메서드 유지 (예: `/api/approvals/{id}/process` 유지, `/submit`·`/approve`로 쪼개지 않음)
- **public 메서드 계약 변경 금지** - 시그니처·이름 유지 (예: `processApproval(id, userId, action, reason)` 유지, 내부는 submit/approve로 위임만)
- **요청/응답 형식 변경 금지** - 기존 request/response 구조 그대로
- **DB 저장값 변경 금지** - enum을 써도 DB엔 기존 정수(0/1/2/3/9 등) 그대로 저장. `@Convert` + `AttributeConverter`로 code 매핑. **`@Enumerated(STRING)`(문자 저장)·`@Enumerated(ORDINAL)`(순번 저장, 9와 어긋남) 금지**
- **동작(observable behavior) 변경 금지** - 조용히 무시하던 케이스를 예외로 바꾸지 않음. 상태 전이 결과·응답·에러 동작을 레거시와 동일하게
- **검증** - 리팩토링 전 특성화 테스트가 리팩토링 후에도 **변경 없이 그대로 green** (호출·기대값·DB값 동일)

## 보존 대상 vs 개선 대상 (결함이 있어도 리팩토링 단계에선 보존)
- 레거시의 나쁜 설계(예: 권한 없는 승인을 **조용히 무시**)도 **리팩토링 단계에선 그대로 보존**한다.
- 특성화 테스트는 "현재 레거시가 그렇게 동작한다"는 **사실을 고정**할 뿐, 옳고 그름을 판단하지 않는다.
- 결함을 바로 고치면 리팩토링이 아니라 **동작 변경**이다 → 리팩토링 완료 후 **별도 단계(재설계)** 에서 다룬다.
- 코드 주석·문서에는 **보존 대상**(동작·계약·DB값)과 **개선 대상**(설계 결함, 차후 재설계)을 명확히 구분해 적는다.

## 리팩토링 주석 규칙 (학습 프로젝트 필수)
- 리팩토링한 코드는 **어떻게 바뀌었는지 반드시 상세 주석으로 남긴다.**
- 클래스 상단: `[리팩토링] 레거시 X → 여기서 Y로 (기법 · 이유)` 헤더 주석
- 변경 지점 인라인: 예 `// [상태 전이 가드] DRAFT일 때만 상신 - 위반 시 조용히 무시(boolean false, 레거시 동작 보존)`
- 목적: 이건 실 프로젝트가 아니라 **학습 산출물** - 코드만 봐도 "레거시에서 무엇을·어떤 기법으로·왜 고쳤는지"가 드러나야 한다.
