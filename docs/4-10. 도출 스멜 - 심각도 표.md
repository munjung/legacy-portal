# 4-10. 도출 스멜 - 심각도 표

`ApprovalService` 클래스 주석에 명시된 11가지 스멜을 [4-4](./4-4.%20코드베이스%20구조%20파악.md)~[4-9](./4-9.%20강결합%20탐지.md)에서 실제로 확인한 근거를 바탕으로 심각도를 매긴다. 이 문서는 지금까지의 탐지 결과를 **종합·우선순위화**하는 것이 목적이며, 새로운 탐지나 리팩토링은 하지 않는다.

## 심각도 판단 기준

세 가지 축으로 심각도를 판단했다.

- **범위**: 몇 개 도메인·몇 곳의 코드에 퍼져 있는가 (넓을수록 심각).
- **위험도**: 방치했을 때 버그로 이어질 가능성이 있는가, 아니면 가독성 문제에 그치는가.
- **선행 조건 여부**: 이 스멜을 안 고치면 다른 스멜을 안전하게 고칠 수 없는가(예: 테스트 부재는 모든 리팩토링의 전제조건).

이 세 축을 종합해 **치명적 > 상 > 중 > 하** 4단계로 분류했다.

## 심각도 표

| 스멜 | 발견 위치(개수) | 심각도 | 판단 근거 | 관련 문서 |
|---|---|---|---|---|
| No Tests | 프로젝트 전체 (테스트 코드 0개) | **치명적** | 안전망이 전혀 없어 지금부터 다루는 모든 스멜을 "고쳐도 되는지" 검증할 방법이 없다. CLAUDE.md도 특성화 테스트를 리팩토링의 선행 조건으로 못박고 있다 — 다른 모든 개선보다 먼저 처리해야 하는 유일한 항목. | - |
| God Class | `ApprovalService` 1곳 (검증·영속화·메일·감사로그·포맷팅·권한판정 6가지 책임 동시 보유) | **상** | 책임이 많을수록 변경 시 영향 범위 예측이 어렵고, 아래 Long Method·Duplicated Code·Feature Envy·Tight Coupling이 전부 이 클래스에 몰려있어 다른 스멜의 근본 원인 역할을 한다. | 4-4, 4-5, 4-7, 4-9 |
| Long Method | `processApproval()` 1곳 — 86줄, 중첩 깊이 6([4-7](./4-7.%20긴%20메서드%20탐지.md)) | **상** | 상신/승인/반려/취소 4개 분기가 한 메서드에 있어 분기 하나를 고치다 다른 분기를 건드릴 위험이 크다. 테스트 없이(No Tests) 이 메서드를 고치는 것이 가장 위험한 조합. | 4-7 |
| Duplicated Code | 감사 로그 6곳·메일 본문 4곳·권한 판정 3곳·직접 `new` 5곳·상태 라벨 2곳 ([4-5](./4-5.%20중복%20코드%20탐지.md), [4-9](./4-9.%20강결합%20탐지.md)) | **상** | 세 서비스에 걸쳐 반복 범위가 가장 넓다. 특히 금액 임계값(100만원)이 두 곳에 별도 리터럴로 존재해([4-8](./4-8.%20매직넘버%20탐지.md) 6번) 하나만 고치면 기준이 어긋나는 실제 버그 위험이 있다. | 4-5, 4-8, 4-9 |
| Tight Coupling | `SmtpMailSender`/`FileAuditLogger` 직접 `new` 5곳 ([4-9](./4-9.%20강결합%20탐지.md)) | **중** | 테스트 작성(치명적 항목 해소)을 가로막는 직접적 원인 — mock 교체가 안 되므로 No Tests 해결의 선결 과제에 가깝다. 다만 협력 클래스 자체가 단순(콘솔 출력)해서 당장 장애를 일으키진 않는다. | 4-9 |
| Magic Number / Primitive Obsession | status(3벌)·action·type·category·role·priority·금액 임계값 전반 ([4-8](./4-8.%20매직넘버%20탐지.md)) | **중** | 범위는 넓지만(★★★ 표 참고) 대부분 `@Convert`+`AttributeConverter`로 기계적으로 흡수 가능한 성격 — God Class·Long Method보다 고치기는 쉬운 편. 단, 100만원 중복 임계값 항목만은 버그 위험이 있어 상 등급에 가깝다. | 4-8 |
| Feature Envy | `ApprovalService.statusLabel()`/`amountGrade()`, `NoticeService.statusLabel()` — 엔티티 데이터를 꺼내 서비스가 직접 규칙 계산 | **중** | 도메인 로직이 서비스에 있어 Rich Domain 전환 시 손대야 할 범위이지만, 순수 함수라 이동 자체는 위험이 낮다. | 4-4, 4-5 |
| Long Parameter List | `ApprovalService.create()` 파라미터 8개 | **하** | 영향 범위가 메서드 1개로 국한. Request DTO 도입 시 자연히 해소되는 항목. | 4-4 |
| Poor Naming | `d`, `u`, `s`, `proc`, `tmp`, `flag1` 등 (`ApprovalService`, `ScheduleService` 전반) | **하** | 가독성 문제일 뿐 동작에는 영향 없음. 다만 CLAUDE.md 네이밍 규칙상 다른 스멜을 고치며 손대는 코드는 예외 없이 함께 개선해야 하는 항목이라 "쉽지만 누락하기 쉬운" 위험이 있다. | 4-4 |
| Comment Smell | 매직넘버·나쁜 이름을 설명하는 주석 다수 | **하** | 독립적인 문제가 아니라 Magic Number·Poor Naming의 증상 — 원인 두 가지가 해소되면 자동으로 사라진다. | 4-8 |

## 요약 — 손대는 순서에 대한 시사점

이 표 자체는 우선순위를 "결정"하는 문서가 아니라 지금까지의 근거를 종합한 것이지만, 심각도 분류에서 자연히 드러나는 순서가 있다.

1. **No Tests(치명적)** 해소 없이는 God Class·Long Method 등 상위 항목을 건드리는 것 자체가 위험하다 — 특성화 테스트가 선행되어야 한다.
2. 상 등급 셋(God Class·Long Method·Duplicated Code)은 전부 `ApprovalService` 한 곳에 몰려 있어, 이 클래스를 다룰 때 함께 해소되는 경우가 많다.
3. Tight Coupling은 그 자체의 위험도는 중이지만 No Tests 해소의 선결 조건이라는 점에서 실질적 우선순위가 높다.
4. Magic Number/Feature Envy는 범위는 넓지만 개별 위험은 낮아 기계적으로 처리 가능한 항목이 대부분이다.
5. 하 등급(Long Parameter List·Poor Naming·Comment Smell)은 위 항목들을 고치는 과정에서 자연히 함께 해소될 가능성이 높은, 부수적 개선 대상이다.

실제 리팩토링 순서·방식은 이 문서가 아니라 별도 단계에서 CLAUDE.md 불변 규칙(API·시그니처·DB 저장값·관찰 가능한 동작 보존)을 지키며 결정한다.
