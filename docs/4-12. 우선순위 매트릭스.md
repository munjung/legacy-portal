# 4-12. 우선순위 매트릭스

[4-10](./4-10.%20도출%20스멜%20-%20심각도%20표.md)의 영향도와 [4-11](./4-11.%20심각도%20스코어링.md)의 영향도×수정난이도 매트릭스에 **예상 공수**를 더해, 실제로 "몇 번째 주에 무엇부터 착수할지" 순서를 정한다. 이 문서가 백로그(다음 단계) 바로 앞 단계다. 세로축 = 영향도, 가로축 = 수정난이도·예상 공수.

## 가로축 정의 — 수정난이도·예상 공수

| 구분 | 난이도 | 예상 공수 | 의미 |
|---|---|---|---|
| S | 하 | 0.5일 이내 | 단순 치환·주입 전환 수준, 실수 여지 거의 없음 |
| M | 중 | 1~2일 | 기계적이지만 빠짐없이 챙겨야 할 지점이 여럿 |
| L | 상 | 3일 이상 | 로직이 얽혀 있어 계획적으로 나눠 진행해야 함 |

## 스멜별 영향도·난이도·공수

| 스멜 | 영향도(4-10) | 난이도(4-11) | 예상 공수 | 구분 |
|---|---|---|---|---|
| No Tests | 치명적 | 중 | 2~3일 (approval/notice/schedule 특성화 테스트 전체) | M |
| God Class | 상 | 상 | 3~4일 (`ApprovalService` 책임 분리) | L |
| Long Method | 상 | 상 | 1~2일 (`processApproval` 4분기 추출 — God Class 작업의 일부로 이어짐) | L |
| Duplicated Code | 상 | 중 | 1~2일 (감사로그·메일·권한판정 공통 모듈 추출) | M |
| Tight Coupling | 중 | 하 | 0.5일 (생성자 주입 전환 5곳) | S |
| Magic Number / Primitive Obsession | 중 | 중 | 2일 (7개 필드 × enum + Converter) | M |
| Feature Envy | 중 | 하 | 0.5일 (`statusLabel`/`amountGrade` 엔티티 이동) | S |
| Long Parameter List | 하 | 하 | 0.5일 (Request DTO 도입) | S |
| Poor Naming | 하 | 하 | 별도 공수 없음 — 위 작업들을 진행하며 함께 처리 | S |
| Comment Smell | 하 | 하 | 별도 공수 없음 — Magic Number/Poor Naming 해소 시 자동 소멸 | S |

## 착수 우선순위 로드맵

공수를 고려하면 "영향도 상"을 무작정 먼저 몰아서 하기보다, **짧은 작업부터 확보해 안전망을 갖춘 뒤** 큰 작업(God Class/Long Method)에 들어가는 순서가 나온다.

| 단계 | 작업 | 공수 | 누적 공수 | 비고 |
|---|---|---|---|---|
| 1단계 | No Tests 해소 (특성화 테스트) | 2~3일 | 2~3일 | 이후 모든 단계의 전제조건 — 반드시 먼저 |
| 2단계 | Tight Coupling 해소 (생성자 주입) | 0.5일 | 2.5~3.5일 | 1단계에서 mock 기반 테스트를 쓰려면 사실상 1단계와 병행/직전 필요 |
| 3단계 | Duplicated Code 해소 (공통 모듈 추출) | 1~2일 | 3.5~5.5일 | 100만원 임계값 중복 등 버그 위험 지점 우선 제거 |
| 4단계 | God Class·Long Method 해소 (`ApprovalService` 책임 분리 + `processApproval` 분기 추출) | 3~4일 | 6.5~9.5일 | 가장 큰 단일 작업 — 1~3단계로 확보한 테스트·공통모듈 위에서 진행 |
| 5단계 | Magic Number/Primitive Obsession 해소 (enum + Converter) | 2일 | 8.5~11.5일 | DB 저장값 불변 규칙 준수하며 도메인별 개별 적용 |
| 6단계 | Feature Envy 해소 (`statusLabel`/`amountGrade` 이동) | 0.5일 | 9~12일 | 4단계 리팩토링과 자연스럽게 이어짐 |
| 7단계 | 잔여 정리 (Long Parameter List·Poor Naming·Comment Smell) | 1일 내외 | 10~13일 | 위 단계들을 진행하며 손대는 코드에 한해 함께 정리, 별도 일정 배정 불필요 |

## 4-11 매트릭스와 달라진 점

- 4-11은 "영향도 상 안에서 난이도로만" 순서를 나눴지만(Duplicated Code → God Class/Long Method), 여기서는 **2단계(Tight Coupling)를 1단계(No Tests) 앞이 아니라 사실상 동시 진행**으로 배치했다 — 테스트에서 mail/audit을 mock으로 바꾸려면 생성자 주입이 먼저 되어 있어야 특성화 테스트 작성이 수월해지기 때문. 공수(0.5일)가 작아 전체 일정에 미치는 영향은 미미하다.
- God Class와 Long Method를 하나의 4단계로 묶었다 — 같은 클래스, 같은 코드 영역이라 실제로는 분리해서 순서를 매길 실익이 없다.
