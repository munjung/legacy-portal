# 4-4. 코드베이스 구조 파악

legacy-portal 의 패키지 구조와 클래스별 책임을 정리한다. 리팩토링에 들어가기 전, "지금 뭐가 있고 무엇을 하는지"를 먼저 고정해두기 위한 문서다.

## 1. 전체 구조

```
com.ktds.portal
├── PortalApplication      - Spring Boot 진입점
├── DataSourceConfig       - DB 자동 선택(MariaDB 우선, 안되면 H2 폴백)
├── DataSeeder             - 시작 시 데모 사용자 4명 시딩
├── common/                - 도메인에 속하지 않는 공통 협력 객체
│   ├── SmtpMailSender     - 메일 발송(실습용 콘솔 출력)
│   └── FileAuditLogger    - 감사 로그 기록(실습용 콘솔 출력)
├── user/                  - 사용자 도메인
│   ├── User               (Entity)
│   ├── UserRepository      (JPA)
│   └── UserController      (REST)
├── approval/              - 결재 도메인
│   ├── Approval            (Entity)
│   ├── ApprovalRepository  (JPA)
│   ├── ApprovalService     (비즈니스 로직)
│   └── ApprovalController  (REST)
├── notice/                - 공지 도메인
│   ├── Notice              (Entity)
│   ├── NoticeRepository    (JPA)
│   └── NoticeService       (비즈니스 로직, Controller 없음 - 미노출)
└── schedule/               - 일정 도메인
    ├── Schedule             (Entity)
    ├── ScheduleRepository   (JPA)
    └── ScheduleService      (비즈니스 로직, Controller 없음 - 미노출)
```

패키지 = 계층(controller/service/…)이 아니라 **도메인(업무 폴더)** 기준으로 나뉜다 (`approval`/`notice`/`schedule`/`user`). 각 도메인 폴더 안에서 Entity + Repository + Service (+ Controller)가 함께 존재한다.

## 2. 도메인별 책임

### 2-1. user (사용자)
- **User**: `role`(1=사원, 2=팀장, 3=임원)을 int 로 저장. 다른 도메인 서비스의 권한 판정(`role >= 2`)에 쓰이는 매직넘버의 근원.
- **UserRepository**: 기본 CRUD만 제공, 커스텀 쿼리 없음.
- **UserController** (`/api/users`): 전체 사용자 목록 조회 1개 엔드포인트. 데모 프론트엔드의 "로그인(사용자 선택)" 화면용.
- Service 계층 없음 - 조회만 하므로 Repository를 Controller가 직접 사용.

### 2-2. approval (결재) — 이 프로젝트의 핵심 실습 대상
- **Approval**: `type`(1=지출/2=휴가/3=구매/4=기타), `status`(0=임시저장/1=상신/2=승인/3=반려/9=취소), `priority`(1=낮음/2=보통/3=높음) 를 모두 int 로 저장. 모든 필드 public setter.
- **ApprovalRepository**: `findByDrafterId`, `findByApproverId` 커스텀 쿼리 제공.
- **ApprovalService**: 결재 생성·상태 전이(상신/승인/반려/취소)·상태 라벨링·금액 등급 계산까지 전부 담당하는 God Class.
  - `create(title, content, type, priority, drafterId, approverId, amount, urgent)` — 결재 생성 + 감사 로그
  - `processApproval(id, userId, action, reason)` — action(1=상신/2=승인/3=반려/9=취소) 하나로 4가지 상태 전이를 전부 분기. 각 분기마다 상태 가드 체크 + 권한 체크(role>=2) + 메일 발송 + 감사 로그가 반복(복붙)됨.
  - 상태 전이 규칙: 상신은 임시저장(0)에서만, 승인/반려는 상신(1) + 본인이 결재자 + role>=2 일 때만, 취소는 임시저장/상신 상태에서 기안자 본인만.
  - 권한 없는 승인 시도, 잘못된 상태에서의 액션 등은 **예외 없이 조용히 무시**(레거시 동작 — 보존 대상).
  - `statusLabel(Approval)` / `amountGrade(Approval)` — 화면 표시용 문자열 변환까지 서비스가 담당(Feature Envy).
  - 협력 객체(`SmtpMailSender`, `FileAuditLogger`)를 생성자 주입이 아니라 필드에서 직접 `new` (강결합).
- **ApprovalController** (`/api/approvals`): `POST /` (생성), `POST /{id}/process` (action 코드로 상신/승인/반려/취소 처리), `GET /drafts/{userId}`, `GET /inbox/{userId}`. 요청 바디를 `Map<String,Object>`로 받아 그대로 캐스팅해 서비스에 전달(요청 DTO 없음, 검증 없음).

### 2-3. notice (공지)
- **Notice**: `status`(0=임시/1=게시/9=내림), `category`(1=일반/2=긴급/3=인사) 를 int 로 저장. `pinned`(boolean)로 상단 고정 여부 표시.
- **NoticeRepository**: 커스텀 쿼리 없음, 기본 CRUD만.
- **NoticeService**: `create(title, content, category, writerId, pinned)`, `publish(id, userId)`(role>=2 + 임시(0) 상태일 때만 게시, 긴급(category==2)이면 전 직원 메일 발송), `statusLabel(Notice)`. **ApprovalService와 거의 동일한 감사 로그/메일 본문/상태 라벨 로직이 그대로 복붙**되어 있음 — 공통 모듈 추출 실습의 주 대상.
- Controller 없음 — REST로 아직 노출되지 않은 도메인.

### 2-4. schedule (일정)
- **Schedule**: `status`(0=예정/1=확정/9=취소) 를 int 로 저장.
- **ScheduleRepository**: `findByOwnerId` 커스텀 쿼리 제공(시간 겹침 검사에 사용).
- **ScheduleService**: `create(title, ownerId, startAt, endAt, location)` — 제목/시간 유효성 검사 + 같은 소유자의 기존 일정과 시간 겹침 여부를 루프로 직접 검사(`flag1` 변수) 후 저장. `confirm(id, userId)` — 본인 소유 + 예정(0) 상태일 때만 확정(1) 처리.
- Controller 없음 — REST로 아직 노출되지 않은 도메인.

### 2-5. common (공통 협력 객체)
- **SmtpMailSender**: 실제로는 발송하지 않고 콘솔에 출력만 하는 실습용 메일 발송기. 인터페이스 없이 구체 클래스로 존재, 각 서비스가 직접 `new` 해서 사용.
- **FileAuditLogger**: 실제로는 파일 기록 대신 콘솔에 출력만 하는 실습용 감사 로그 기록기. 마찬가지로 인터페이스 없이 각 서비스가 직접 `new`.

### 2-6. 애플리케이션 레벨
- **PortalApplication**: `@SpringBootApplication` 진입점, `main()`만 존재.
- **DataSourceConfig**: 시작 시 `localhost:3306/portal` MariaDB 연결을 2초 타임아웃으로 시도 → 성공하면 MariaDB, 실패하면 H2 인메모리(`jdbc:h2:mem:portal`, LEGACY 모드)로 자동 폴백하는 `DataSource` 빈을 만든다. 별도 프로파일 전환 없이 실습 환경에 따라 자동 동작.
- **DataSeeder**: `CommandLineRunner`. 앱 시작 시 `UserRepository`가 비어 있으면 데모 사용자 4명(김사원/박팀장/이임원/최사원, role 1~3 섞임)을 저장.

## 3. 도메인 간 의존 관계

- `approval`, `notice`, `schedule` 서비스는 모두 `user.UserRepository`에 의존(권한/소유자 조회).
- `approval`, `notice` 서비스는 `common.SmtpMailSender` + `common.FileAuditLogger`에 의존(직접 `new`, DI 아님).
- `schedule` 서비스는 `common.FileAuditLogger`에만 의존(메일 발송 없음).
- 도메인 간 직접 참조(예: approval → notice)는 없음 — 전부 `user`를 통해서만 교차.
- REST로 노출된 도메인은 `approval`, `user` 뿐이고 `notice`, `schedule`은 Service까지만 존재(Controller 미구현 상태).

## 4. 의도적으로 심어둔 코드 스멜 (실습 포인트)

`ApprovalService` 클래스 상단 주석에 명시된 11가지가 이 프로젝트 전체를 관통하는 실습 소재다:

1. God Class — ApprovalService가 검증/영속화/메일/감사로그/포맷팅/권한판정을 혼자 담당
2. Long Method — `processApproval()` 100줄+, 중첩 if 6단계
3. Magic Number — status/type/role/priority/category 가 전부 int 리터럴
4. Duplicated Code — 메일 본문 생성, 감사 로그 기록이 `approval`/`notice`/`schedule` 서비스마다 복붙
5. Tight Coupling — `new SmtpMailSender()`, `new FileAuditLogger()` 직접 생성(DI 없음)
6. Feature Envy — 서비스가 엔티티 필드를 꺼내 상태/금액 규칙을 직접 계산
7. Primitive Obsession — 모든 분기를 int 비교로 처리
8. Long Parameter List — `ApprovalService.create()` 파라미터 8개
9. Poor Naming — `d`, `u`, `s`, `proc`, `tmp`, `flag1` 같은 약어
10. Comment Smell — 나쁜 이름/매직넘버를 주석으로 변명
11. No Tests — 테스트 코드 전무(안전망 부재)

리팩토링 시 CLAUDE.md의 [불변 규칙](../CLAUDE.md)에 따라 API 엔드포인트, public 메서드 시그니처, 요청/응답 구조, DB 저장값(int), 조용히 무시하는 동작은 그대로 유지하고 **내부 구조만** 개선한다.
