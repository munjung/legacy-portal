# 4-5. 중복 코드 탐지

`ApprovalService` / `NoticeService` / `ScheduleService` 세 서비스에서 반복되는 코드를 찾아 정리한다. [4-4. 코드베이스 구조 파악](./4-4.%20코드베이스%20구조%20파악.md)에서 언급한 "스멜4 Duplicated Code"의 실제 위치를 전부 짚는다. 문서만 정리하는 단계이며 코드는 아직 건드리지 않는다 — 리팩토링 여부/방식은 CLAUDE.md 불변 규칙에 따라 별도 단계에서 결정한다.

## 1. 감사 로그(Audit Log) 기록 — 완전 동일 패턴, 6곳

타임스탬프를 `yyyy-MM-dd HH:mm:ss`로 포맷해서 `"[시각] 액션 id=... by=..."` 형태 문자열을 만들고 `audit.write(...)`에 넘기는 코드가 세 서비스에 걸쳐 6곳에 나온다.

| 위치 | 코드 |
|---|---|
| `ApprovalService.create()` (`ApprovalService.java:62-65`) | `String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`String line = "[" + now + "] APPROVAL CREATE id=" + d.getId() + " by=" + drafterId + " type=" + d.getType();`<br>`audit.write(line);` |
| `ApprovalService.writeAudit()` (`ApprovalService.java:163-166`, SUBMIT/APPROVE/REJECT/CANCEL 4곳에서 호출) | `String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`audit.write("[" + now + "] " + act + " id=" + id + " by=" + userId);` |
| `NoticeService.create()` (`NoticeService.java:44-45`) | `String now = ...ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`audit.write("[" + now + "] NOTICE CREATE id=" + n.getId() + " by=" + writerId);` |
| `NoticeService.publish()` (`NoticeService.java:69-70`) | `String now = ...ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`audit.write("[" + now + "] NOTICE PUBLISH id=" + n.getId() + " by=" + userId);` |
| `ScheduleService.create()` (`ScheduleService.java:61-63`) | `String now = ...ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`audit.write("[" + now + "] SCHEDULE CREATE id=" + sc.getId() + " by=" + ownerId);` |
| `ScheduleService.confirm()` (`ScheduleService.java:76-77`) | `String now = ...ofPattern("yyyy-MM-dd HH:mm:ss"));`<br>`audit.write("[" + now + "] SCHEDULE CONFIRM id=" + sc.getId() + " by=" + userId);` |

- `ApprovalService`는 자체적으로 `writeAudit()` 헬퍼로 부분 추출까지 했지만, `create()` 안에는 여전히 복붙이 남아있다("불완전한 중복 제거" — `ApprovalService.java:162` 주석에도 명시).
- `NoticeService`, `ScheduleService`는 추출조차 하지 않고 매번 인라인으로 반복.
- 공통점: **타임스탬프 포맷팅 로직**(`DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss")`)과 **"[시각] 액션 id=X by=Y" 문자열 조립 규칙**이 도메인과 무관하게 완전히 동일.

## 2. 메일 본문 생성 + 발송 — 거의 동일 패턴, 4곳

인사말 + 안내 문구 + 제목을 조합해 본문을 만들고 `mail.send(수신자 이메일, "[라벨] 제목", body)`를 호출하는 코드가 4곳.

| 위치 | 코드 |
|---|---|
| `ApprovalService.processApproval()` SUBMIT (`ApprovalService.java:103-106`) | `"안녕하세요 " + approver.getName() + "님,\n" + "결재 요청이 도착했습니다.\n제목: " + d.getTitle() + "\n기안자ID: " + d.getDrafterId()` → `mail.send(approver.getEmail(), "[결재요청] " + d.getTitle(), body)` |
| `ApprovalService.processApproval()` APPROVE (`ApprovalService.java:121-123`) | `"안녕하세요 " + drafter.getName() + "님,\n" + "결재가 승인되었습니다.\n제목: " + d.getTitle()` → `mail.send(drafter.getEmail(), "[결재승인] " + d.getTitle(), body)` |
| `ApprovalService.processApproval()` REJECT (`ApprovalService.java:140-143`) | `"안녕하세요 " + drafter.getName() + "님,\n" + "결재가 반려되었습니다.\n제목: " + d.getTitle() + "\n사유: " + reason` → `mail.send(drafter.getEmail(), "[결재반려] " + d.getTitle(), body)` |
| `NoticeService.publish()` 긴급공지 (`NoticeService.java:64-66`) | `"안녕하세요 " + member.getName() + "님,\n" + "긴급 공지가 게시되었습니다.\n제목: " + n.getTitle()` → `mail.send(member.getEmail(), "[긴급공지] " + n.getTitle(), body)` |

- 공통 골격: `"안녕하세요 " + 수신자.getName() + "님,\n" + <안내 문구> + "\n제목: " + 대상.getTitle() [+ 선택적 추가 줄]`, 그리고 `mail.send(수신자.getEmail(), "[라벨] " + 대상.getTitle(), body)`.
- 차이는 안내 문구·라벨·수신자(결재자/기안자/전 직원)뿐 — 파라미터화하면 하나의 메서드로 합칠 수 있는 전형적 중복.
- `ScheduleService`에는 메일 발송이 없어 이 패턴에서 제외.

## 3. 권한 판정(`role >= 2`) — 완전 동일 조건식, 3곳

"팀장 이상만 가능"이라는 동일한 권한 규칙이 그대로 반복된다.

| 위치 | 코드 |
|---|---|
| `ApprovalService.processApproval()` APPROVE (`ApprovalService.java:114`) | `if (u.getRole() >= 2) {` |
| `ApprovalService.processApproval()` REJECT (`ApprovalService.java:133`) | `if (u.getRole() >= 2) {` |
| `NoticeService.publish()` (`NoticeService.java:56`) | `if (u.getRole() >= 2) {` |

- `User.role`: 1=사원·2=팀장·3=임원 (`User.java:19`). "팀장 이상"이라는 의미가 코드 어디에도 이름으로 남지 않고 `>= 2`라는 리터럴로만 3곳에 흩어져 있음.

## 4. 상태(status) → 표시 라벨 변환 — 구조적 중복, 2곳

값은 다르지만 **모양이 완전히 동일한** if/else 사다리로 int 상태 코드를 한글 라벨 문자열로 바꾼다.

| 위치 | 코드 |
|---|---|
| `ApprovalService.statusLabel()` (`ApprovalService.java:169-179`) | `s==0→"임시저장", s==1→"상신", s==2→"승인", s==3→"반려", s==9→"취소", else→"알수없음"` |
| `NoticeService.statusLabel()` (`NoticeService.java:76-82`) | `s==0→"임시", s==1→"게시", s==9→"내림", else→"알수없음"` |

- 두 메서드 모두 "status getter → if/else 사다리 → 알수없음 폴백" 구조가 동일하다. 이는 코드 텍스트 자체는 다르지만(값·라벨 다름), **패턴이 반복**되는 구조적 중복이며 도메인별 status enum이 생기면 `enum.getLabel()` 형태로 자연스럽게 흡수될 대상이다.
- `ScheduleService`에는 아직 `statusLabel()`이 없지만 `Schedule.status`도 동일한 0/1/9 체계(예정/확정/취소)를 쓰고 있어, 추가된다면 같은 패턴이 세 번째로 반복될 가능성이 높다.

## 5. "조회 후 없으면 조용히 무시" 가드 — 구조적 중복, 여러 곳

ID로 엔티티/사용자를 조회해 없으면 아무 예외 없이 조용히 `return`하는 가드가 세 서비스 모두에 반복된다.

| 위치 | 코드 |
|---|---|
| `ApprovalService.processApproval()` (`ApprovalService.java:76-84`) | `Approval d = repo.findById(id).orElse(null); if (d == null) return;` / `User u = userRepo.findById(userId).orElse(null); if (u == null) return;` |
| `NoticeService.publish()` (`NoticeService.java:50-53`) | `Notice n = repo.findById(id).orElse(null); if (n == null) return;` / `User u = userRepo.findById(userId).orElse(null); if (u == null) return;` |
| `ScheduleService.confirm()` (`ScheduleService.java:68-71`) | `Schedule sc = repo.findById(id).orElse(null); if (sc == null) return;` / `User u = userRepo.findById(userId).orElse(null); if (u == null) return;` |

- **주의(보존 대상)**: "존재하지 않으면 예외 없이 조용히 무시"는 CLAUDE.md가 명시한 레거시의 관찰 가능한 동작이다. 중복을 걷어내더라도 이 동작 자체(호출자가 실패를 알 수 없는 것)는 리팩토링 단계에서 그대로 유지해야 한다.

## 6. 협력 객체 직접 `new` — 동일한 강결합 패턴, 3곳

DI 없이 서비스가 필드에서 협력 객체를 직접 생성하는 동일한 패턴.

| 위치 | 코드 |
|---|---|
| `ApprovalService` (`ApprovalService.java:37-38`) | `private final SmtpMailSender mail = new SmtpMailSender();`<br>`private final FileAuditLogger audit = new FileAuditLogger();` |
| `NoticeService` (`NoticeService.java:24-25`) | `private final SmtpMailSender mail = new SmtpMailSender();`<br>`private final FileAuditLogger audit = new FileAuditLogger();` |
| `ScheduleService` (`ScheduleService.java:20`) | `private final FileAuditLogger audit = new FileAuditLogger();` |

- 세 서비스 모두 생성자 주입 대신 필드 초기화에서 직접 `new` — 테스트에서 목(mock)으로 교체 불가능한 동일한 구조가 반복.

## 요약

| 중복 유형 | 반복 횟수 | 성격 |
|---|---|---|
| 감사 로그 타임스탬프+문자열 조립 | 6 | 완전 동일(리터럴 중복) |
| 메일 본문 생성 + 발송 | 4 | 거의 동일(파라미터만 다름) |
| 권한 판정 `role >= 2` | 3 | 완전 동일 조건식 |
| 협력 객체 직접 `new` | 3 (서비스 3곳) | 완전 동일 구조 |
| status → 라벨 변환 | 2 (+ 잠재적으로 Schedule까지 3) | 구조적 동일, 값만 다름 |
| 조회 후 없으면 조용히 무시 | 3+ (서비스마다 반복) | 구조적 동일 — **동작은 보존 대상** |

가장 유력한 리팩토링 방향은 CLAUDE.md의 "직접 `new` → 생성자 주입(DIP)"과 "공통 모듈 추출"이다: 감사 로그·메일 발송 조립 로직을 `common` 패키지의 협력 클래스(또는 각 서비스가 주입받는 헬퍼)로 추출하면 1·2·6번 중복이 한 번에 해소된다. 다만 추출 시에도 `audit.write(...)`에 최종적으로 들어가는 로그 문자열, 메일 제목/본문, 저장되는 DB 값(int)은 레거시와 **글자 하나까지 동일하게** 유지해야 특성화 테스트가 깨지지 않는다.
