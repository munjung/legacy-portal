# 4-7. 긴 메서드 탐지

전체 Java 소스(`approval`/`notice`/`schedule`/`user`/`common` + 애플리케이션 레벨)를 대상으로 메서드별 **라인 수**와 **중첩 깊이**를 조사했다. Getter/Setter, 1~3줄짜리 위임 메서드, Repository 인터페이스처럼 분기 로직이 없는 코드는 표에서 제외했다(전부 세어도 의미 있는 신호가 안 나옴).

## 측정 기준

- **라인 수**: 메서드 시그니처 줄부터 닫는 `}` 줄까지 포함한 줄 수.
- **중첩 깊이**: 메서드 바디 자체를 **깊이 1**로 놓고, 그 안에 `if`/`for`/`while`/`try` 같은 제어 블록이 한 겹 들어갈 때마다 +1. `else if`는 형제(sibling) 분기라 깊이를 올리지 않는다(같은 조건 사다리 안에서만 갈라짐). `ApprovalService` 클래스 주석의 "중첩 if 6단계" 표현과 동일한 셈법이다.

## `processApproval()` 상세 분해 — 유일한 50줄 초과 메서드

`ApprovalService.java:75-160`, 총 86줄. `action`(1=상신·2=승인·3=반려·9=취소) 값 하나로 4가지 상태 전이를 전부 처리하는 하나의 메서드다.

| 분기 | 라인 범위 | 줄 수 | 중첩 깊이 | 깊이가 쌓이는 경로 |
|---|---|---:|---:|---|
| 공통 가드(조회 실패 시 조용히 리턴) | 76-84 | 9 | 2 | `if (d == null)`, `if (u == null)` |
| `action==1` 상신 | 90-109 | 20 | 4 | `if(proc==1)` → `if(s==0)` → `if(type==1 && amount>=1e6)` / `if(approver!=null)` |
| `action==2` 승인 | 110-128 | 19 | **6** | `if(proc==2)` → `if(s==1)` → `if(approverId.equals)` → `if(role>=2)` → `if(drafter!=null)` |
| `action==3` 반려 | 129-148 | 20 | **6** | `if(proc==3)` → `if(s==1)` → `if(approverId.equals)` → `if(role>=2)` → `if(drafter!=null)` |
| `action==9` 취소 | 149-159 | 11 | 4 | `if(proc==9)` → `if(s==0\|\|s==1)` → `if(drafterId.equals)` |

승인·반려 분기가 각각 깊이 6까지 들어가는 이유는 "상태 확인 → 결재자 본인 확인 → 권한(role>=2) 확인 → 처리 → 메일 대상 존재 확인"이 전부 순차적으로 중첩된 `if`로 쌓여 있기 때문이다. 승인/반려 두 분기는 조건 구조가 사실상 동일해서(`4-5. 중복 코드 탐지.md`의 권한 판정 중복과 같은 지점) 깊이도 나란히 6으로 같다.

## 참고

이 문서는 현재 상태를 있는 그대로 기록한 것으로, 리팩토링 여부·방식은 다루지 않는다. CLAUDE.md의 "긴 메서드 분리·로직을 도메인으로 이동(Rich Domain)" 방향과 연결지어 보면, `processApproval()`의 4개 분기(상신/승인/반려/취소)를 메서드로 추출하는 것이 가장 먼저 눈에 띄는 후보이지만, 추출 이후에도 **`processApproval(id, userId, action, reason)`라는 public 시그니처와 각 분기의 관찰 가능한 동작(조용히 무시하는 케이스 포함)은 CLAUDE.md 불변 규칙에 따라 그대로 유지**해야 한다.
