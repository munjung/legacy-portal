# 4-8. 매직넘버 탐지

코드 전체에서 정수 리터럴로 흩어져 있는 "의미가 코드에 없고 주석에만 있는" 매직넘버를 찾아, 각 숫자가 실제로 무엇을 뜻하는지 정리한다. 문서만 정리하는 단계이며, enum화 등 실제 리팩토링은 아직 하지 않는다. CLAUDE.md 불변 규칙상 **DB 저장값(정수)은 리팩토링 후에도 그대로 유지**해야 하므로, 여기서 추정한 의미는 향후 `@Convert` + `AttributeConverter` 매핑에 그대로 쓰일 근거 자료가 된다.

## 1. 상태(status) 코드 — 도메인별 3벌

같은 "0/1/…/9" 체계를 세 도메인이 각자 따로 정의해서 쓰고 있고, 숫자가 같아도 의미는 도메인마다 다르다.

| 도메인 | 필드 | 값 | 추정 의미 | 근거 위치 |
|---|---|---|---|---|
| 결재(Approval) | `status` | 0 | 임시저장 (DRAFT) | `Approval.java:26`, `ApprovalService.java:53,92` |
| | | 1 | 상신 (SUBMITTED) | `ApprovalService.java:97,111` |
| | | 2 | 승인 (APPROVED) | `ApprovalService.java:115` |
| | | 3 | 반려 (REJECTED) | `ApprovalService.java:134` |
| | | 9 | 취소 (CANCELED) | `ApprovalService.java:150,153` |
| 공지(Notice) | `status` | 0 | 임시(게시 전) | `Notice.java:19`, `NoticeService.java:37,57` |
| | | 1 | 게시 (PUBLISHED) | `NoticeService.java:58` |
| | | 9 | 내림 (내려짐 — 실제로 이 상태로 전이시키는 코드는 아직 없음, 라벨만 존재) | `NoticeService.java:81` |
| 일정(Schedule) | `status` | 0 | 예정 | `Schedule.java:18`, `ScheduleService.java:58` |
| | | 1 | 확정 (CONFIRMED) | `ScheduleService.java:74` |
| | | 9 | 취소 (CANCELED, 겹침 검사에서 제외됨) | `ScheduleService.java:42` |

- **공통 패턴**: 세 도메인 모두 `9`를 "취소/내림" 같은 종료성 음성 상태에 쓰고, `0`을 "아직 확정 전" 초기 상태에 쓴다. 다만 왜 하필 `9`인지(4~8을 비워둔 이유)는 코드 어디에도 설명이 없다 — `ApprovalService.java:149` 주석이 "왜 9? 4~8은 비워둔 규칙 없는 번호"라고 스스로 지적하는 지점.
- **개선 시 주의**: enum으로 바꿔도 DB에는 이 정수가 그대로 저장되어야 한다(CLAUDE.md 불변 규칙). 세 도메인의 상태는 숫자가 겹치더라도(예: 0, 1, 9) **서로 다른 enum**이어야 한다 — 우연히 값이 같다고 하나의 공용 enum으로 묶으면 의미가 왜곡된다.

## 2. 결재 처리 액션(action) 코드 — API 계약에 노출된 매직넘버

`ApprovalController.process()` / `ApprovalService.processApproval()`이 받는 `action` 파라미터. 위 결재 status와 숫자가 겹치지만 "상태"가 아니라 "요청한 동작"이라는 별개 개념이다.

| 값 | 추정 의미 | 근거 위치 |
|---|---|---|
| 1 | 상신 요청 (SUBMIT) | `ApprovalController.java:36`, `ApprovalService.java:90` |
| 2 | 승인 요청 (APPROVE) | `ApprovalService.java:110` |
| 3 | 반려 요청 (REJECT) | `ApprovalService.java:129` |
| 9 | 취소 요청 (CANCEL) | `ApprovalService.java:147` |

- **주의**: 이 숫자는 `POST /api/approvals/{id}/process` 요청 바디의 `action` 필드로 그대로 노출되어 있다 — 즉 **API 클라이언트가 이 정수를 외워서 보내야 하는 계약의 일부**다. CLAUDE.md 불변 규칙상 요청/응답 형식은 바꿀 수 없으므로, enum화하더라도 클라이언트가 보내는 JSON의 `action` 값(1/2/3/9)은 그대로 받아들여야 한다.

## 3. 분류(type/category) 코드

| 도메인 | 필드 | 값 | 추정 의미 | 근거 위치 |
|---|---|---|---|---|
| 결재 | `type` | 1 | 지출 | `Approval.java:25`, `ApprovalService.java:94` (`type==1` && 금액 조건에서 사용) |
| | | 2 | 휴가 | `Approval.java:25` (주석에만 존재, 분기 로직에서 실제로 검사하는 곳 없음) |
| | | 3 | 구매 | 〃 |
| | | 4 | 기타 | 〃 |
| 공지 | `category` | 1 | 일반 | `Notice.java:20` |
| | | 2 | 긴급 (게시 시 전 직원 메일 발송 트리거) | `NoticeService.java:62` |
| | | 3 | 인사 | `Notice.java:20` (주석에만 존재) |

- 결재 `type`은 4가지 값이 정의돼 있지만, 실제 서비스 로직에서 분기하는 것은 `type==1`(지출) 하나뿐이다(금액 자동 상향 규칙). 2/3/4(휴가/구매/기타)는 저장·조회만 될 뿐 서비스 로직상 의미 있는 분기가 없다 — enum화 시에도 "정의는 있으나 로직 미사용"이라는 현재 상태를 그대로 유지해야 한다(동작 변경 금지).
- 공지 `category==2`(긴급)만 실제로 분기(`NoticeService.java:62`)에 쓰이고, 1/3은 라벨/저장 용도로만 존재.

## 4. 우선순위(priority) 코드

| 값 | 추정 의미 | 근거 위치 |
|---|---|---|
| 1 | 낮음 | `Approval.java:27` |
| 2 | 보통 | 〃 (기본값, `ApprovalController.java:28` 요청에서 그대로 전달) |
| 3 | 높음 | `ApprovalService.java:52,95` (`urgent` 플래그 또는 100만원 이상 지출 시 자동 설정) |

- "왜 3이 높음인가"에 대한 설명은 없다(`ApprovalService.java:52` 주석이 스스로 지적). 낮음(1) < 보통(2) < 높음(3)이라는 순서 자체는 자연스럽지만, enum 도입 시 정수 비교(`priority >= 2` 같은 코드는 현재 없음)에 의존하는 곳이 있는지 확인 필요 — 현재는 순서 비교 없이 값 설정만 하고 있어 안전.

## 5. 권한(role) 코드와 권한 판정 임계값

| 값 | 추정 의미 | 근거 위치 |
|---|---|---|
| 1 | 사원 | `User.java:19`, `DataSeeder.java:24,27` |
| 2 | 팀장 | `User.java:19`, `DataSeeder.java:25` |
| 3 | 임원 | `User.java:19`, `DataSeeder.java:26` |
| `role >= 2` | "팀장 이상"에게만 허용되는 권한 임계값 | `ApprovalService.java:114`(승인), `ApprovalService.java:133`(반려), `NoticeService.java:56`(공지 게시) — 3곳에 동일한 리터럴로 반복(`4-5. 중복 코드 탐지.md` 3번 항목과 동일 지점) |

- `role`은 순서가 있는 등급(사원<팀장<임원)이라 단순 `==` 매핑 enum보다는 "등급 비교가 가능한 개념"으로 보인다. `role >= 2`라는 부등호 조건이 3곳에 반복되는 것도 이 값이 "카테고리"가 아니라 "등급"이라는 신호.

## 6. 금액 기준 임계값 — 도메인 규칙에 박힌 매직넘버

`ApprovalService`에 금액을 등급/우선순위로 변환하는 리터럴이 두 군데 따로 존재한다.

| 위치 | 리터럴 | 추정 의미 |
|---|---|---|
| `ApprovalService.java:94` | `1000000` (100만원) | 지출 결재 상신 시, 이 금액 이상이면 우선순위를 자동으로 "높음(3)"으로 상향 |
| `ApprovalService.java:184` | `10000000` (1000만원) | `amountGrade()`: 이 금액 이상이면 등급 "S" |
| `ApprovalService.java:185` | `1000000` (100만원) | `amountGrade()`: 이 금액 이상이면 등급 "A" (위 상신 기준과 같은 100만원이지만 **별개의 상수**로 각자 하드코딩되어 있어, 하나를 고치면 다른 하나는 안 바뀌는 위험이 있음) |
| `ApprovalService.java:186` | `100000` (10만원) | `amountGrade()`: 이 금액 이상이면 등급 "B", 미만이면 "C" |

- "100만원"이라는 동일한 기준값이 상신 우선순위 로직(94번 줄)과 금액 등급 로직(185번 줄)에 **각각 다른 리터럴로 중복 정의**되어 있다는 점이 특히 위험하다 — 나중에 "100만원 기준을 200만원으로 바꿔달라"는 요구가 오면 두 곳을 다 찾아 고쳐야 하는데, 코드만 봐서는 두 곳이 "같은 기준"이라는 사실이 드러나지 않는다.

## 요약 — enum 후보 우선순위

| 필드 | 값 종류 | 실제 로직에서 분기 여부 | enum화 우선순위 |
|---|---|---|---|
| `Approval.status` / action 코드 | 0,1,2,3,9 | 5개 상태 전이 전부 분기 (`processApproval` 중심) | ★★★ 최우선 — `4-7. 긴 메서드 탐지.md`의 processApproval과 직결 |
| `User.role` | 1,2,3 | `role>=2` 부등호 판정 3곳 | ★★★ — 등급 비교 의미를 이름으로 드러낼 수 있음 |
| `Notice.status` | 0,1,9 | 게시 여부 분기 | ★★ |
| `Schedule.status` | 0,1,9 | 확정/겹침제외 분기 | ★★ |
| `Notice.category` | 1,2,3 | `category==2`만 분기 | ★★ (2 외에는 로직 없음, 라벨용) |
| `Approval.priority` | 1,2,3 | 값 설정만, 비교 로직 없음 | ★ |
| `Approval.type` | 1,2,3,4 | `type==1`만 분기 | ★ (1 외에는 로직 없음, 라벨/저장용) |
| 금액 임계값(100만/1000만/10만원) | — | 서비스 메서드 2곳에 중복 하드코딩 | ★★ — enum은 아니지만 이름 있는 상수로 추출할 최우선 후보 (중복 위험 때문에) |

모든 항목에서 **DB에는 지금의 정수값이 그대로 저장되어야 한다**(`@Enumerated(STRING)`/`@Enumerated(ORDINAL)` 금지, `@Convert` + `AttributeConverter`로 code 매핑) — CLAUDE.md 불변 규칙.
