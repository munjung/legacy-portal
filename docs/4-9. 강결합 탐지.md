# 4-9. 강결합 탐지

서비스가 협력 객체(다른 클래스가 제공하는 기능)를 생성자 주입 없이 **직접 `new`로 생성**하는 지점을 찾는다. `ApprovalService` 클래스 주석의 "스멜5 Tight Coupling"이 코드베이스 전체에서 실제로 어디에 있는지 확인하는 문서다. 문서만 정리하는 단계이고, 생성자 주입(DIP) 전환 등 실제 리팩토링은 하지 않는다.

## 조사 범위와 제외 기준

`new`가 쓰인 모든 곳(`DataSeeder`의 `new User(...)`, 각 서비스의 `new Approval()`/`new Notice()`/`new Schedule()` 등)을 살펴봤지만, 이 중 **엔티티(Entity)를 새로 만드는 `new`는 강결합 문제에서 제외**한다. `new Approval()`처럼 매번 새로운 도메인 객체(생성할 대상 그 자체)를 만드는 것은 정상적인 객체 생성이지, 협력자를 갈아끼울 수 없게 고정하는 "의존성"이 아니기 때문이다. 강결합의 대상은 **서비스가 위임하는 외부 능력(메일 발송, 감사 로그 기록 등)을 제공하는 협력 클래스**로 한정한다.

## 발견한 강결합 지점

| 클래스 (필드) | 협력 클래스 | 위치 | 인터페이스 존재 여부 |
|---|---|---|---|
| `ApprovalService.mail` | `SmtpMailSender` | `ApprovalService.java:37` | 없음 (구체 클래스 직접 의존) |
| `ApprovalService.audit` | `FileAuditLogger` | `ApprovalService.java:38` | 없음 |
| `NoticeService.mail` | `SmtpMailSender` | `NoticeService.java:24` | 없음 |
| `NoticeService.audit` | `FileAuditLogger` | `NoticeService.java:25` | 없음 |
| `ScheduleService.audit` | `FileAuditLogger` | `ScheduleService.java:20` | 없음 |

전부 아래와 같은 동일한 형태다:

```java
private final SmtpMailSender mail = new SmtpMailSender();
private final FileAuditLogger audit = new FileAuditLogger();
```

세 서비스 모두 `@Service`로 등록되어 Spring이 관리하는 빈인데도, 정작 자기 필드는 생성자 주입 대신 필드 초기화 시점에 직접 `new` 한다 — 다른 협력자(`ApprovalRepository`, `UserRepository` 등)는 전부 생성자로 주입받으면서 `SmtpMailSender`/`FileAuditLogger`만 예외적으로 직접 생성한다는 점이 눈에 띈다.

## 의존 대상별 재사용 현황

| 협력 클래스 | 직접 `new` 하는 서비스 수 | 목록 |
|---|---|---|
| `FileAuditLogger` | 3 | `ApprovalService`, `NoticeService`, `ScheduleService` (도메인 서비스 전부) |
| `SmtpMailSender` | 2 | `ApprovalService`, `NoticeService` (`ScheduleService`는 메일 발송 없음) |

`FileAuditLogger`는 세 도메인 서비스 모두가 각자 인스턴스를 하나씩 만들어 쓰고 있다 — 즉 동일한 기능을 하는 객체가 서비스 개수만큼(3개) 중복 생성된다.

## 왜 문제인가

- **테스트 불가**: `SmtpMailSender`/`FileAuditLogger` 둘 다 인터페이스가 없는 구체 클래스이고, 서비스가 필드에서 직접 생성하므로 테스트 코드에서 가짜(mock/stub) 구현으로 교체할 방법이 없다. 지금 실제로 발송/기록 대신 콘솔에 출력만 하는 이유도 "테스트·실습 환경에서 실제 SMTP/파일 시스템을 안 써도 되게" 하기 위한 임시 처리로 보이는데, 이마저도 구조상 강제로 실행된다(호출 자체를 막을 수 없음).
- **협력자 교체 불가**: `FileAuditLogger`(콘솔 출력)를 나중에 실제 파일 기록이나 DB 기록으로 바꾸려면, 이를 사용하는 서비스 3곳의 소스 코드를 전부 찾아 고쳐야 한다. 생성자 주입이었다면 구현체만 교체(빈 등록)하면 끝날 일이다.
- **`@Service` 빈인데 절반만 DI를 따름**: 같은 클래스 안에서 `ApprovalRepository`/`UserRepository`는 생성자 주입, `SmtpMailSender`/`FileAuditLogger`는 필드에서 직접 `new` — 일관성이 없어 다음에 필드를 보는 사람이 "이건 왜 주입이 아니지?"라는 의문 없이 그냥 관행처럼 넘어가기 쉽다.

## 참고

CLAUDE.md의 "직접 `new` → 생성자 주입(DIP)" 개선 방향과 정확히 일치하는 지점이다. 다만 이 문서는 탐지 결과만 기록하며, 인터페이스 추출(`MailSender`/`AuditLogger` 같은 추상화 도입 여부 포함)과 실제 생성자 주입 전환은 별도 리팩토링 단계에서 다룬다 — 전환 시에도 `mail.send(...)`/`audit.write(...)` 호출 결과로 콘솔에 찍히는 문자열, 순서, 발생 여부는 레거시와 동일해야 한다(관찰 가능한 동작 불변).
