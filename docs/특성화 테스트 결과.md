## 4. 해석

- **"성공 경로"는 잘 덮여 있다.** 상신→승인, 반려, 취소 각각의 정상 흐름과 `writeAudit`/메일 발송까지는 6개 테스트로 확인됨.
- **가장 큰 공백은 "반려·취소의 권한없음"이다.** 승인 쪽은 4번 테스트(`지정된_결재자가_아닌_사용자가_승인을_시도하면_조용히_무시된다`)로 "다른 사람이 승인 시도"가 검증됐지만, **반려(#9)·취소(#14)의 동일한 권한 검사는 단 한 번도 실행되지 않았다.** 리팩토링 중 반려/취소의 권한 가드를 실수로 빠뜨려도 지금 이 6개 테스트는 잡아내지 못한다.
- **"역할 부족(role<2)"과 "잘못된 사람(다른 userId)"은 서로 다른 분기다.** 4번 테스트는 후자만 검증했고, 전자(#6, #10 — 지정된 결재자 본인이지만 사원 등급이라 승인/반려 권한이 없는 경우)는 승인·반려 양쪽 다 비어 있다.
- **금액 기준 우선순위 자동상향(#3)이 통째로 미검증**이다 — `ApprovalService.java:94-95`의 "지출 100만원 이상이면 priority=3으로 자동 상향"이라는 도메인 규칙은 지금 어떤 테스트도 건드리지 않는다.
- **정의되지 않은 action 코드(#12)** 도 비어 있다 — `processApproval`이 사실상 `switch` 없는 if-elseif 체인이라, 1/2/3/9 외의 값이 들어오면 아무 분기도 안 타고 그냥 끝난다는 "조용한 무시" 동작이 문서화는 됐지만 테스트로 고정되진 않았다.
- **null 참조 계열(#1, #4, #5, #7, #9 일부, #11, #14 일부)** 은 대부분 "FK가 가리키는 대상이 실제로는 없는" 데이터 정합성 케이스로, 우선순위는 낮지만 존재는 한다.
